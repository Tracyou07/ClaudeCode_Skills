# 科研项目管理工具

Claude Code 中用于科研代码开发的进度管理、任务执行和算法检查 Skill 合集。三个 Skill 共享相同的多 Agent 工作模式。

## 触发方式

三个 Skill 均使用**精确触发词**，不会被类似表述误触发：

| Skill | 精确触发词 | 功能 |
|-------|-----------|------|
| `research-progress-tracker` | **科研启动** | 进度追踪 + Git 管理 |
| `research-task-executor` | **科研启动-执行** | 任务执行 |
| `research-algorithm-checker` | **科研启动-检查** | 算法审查 |

## 共同特征

### 多 Agent 工作模式

每个 Skill 启动后使用 **5 个 Agent** 并行工作，每个 Agent 均为 AI + LLM 科研专家，具备：
- 逻辑严谨、不跳步不遗漏的思考过程
- 科研项目管理能力
- 深入研究现有代码库后再动手

## 各 Skill 详情

### 1. 科研启动 — 进度追踪

触发词: `科研启动`

**核心功能**:

| 功能 | 说明 |
|------|------|
| Git 进度管理 | 每完成改动后自动 `git add` + `git commit`，清晰的中英文 commit message |
| PROGRESS.md | 在项目根目录维护代码完成记录文档 |
| 不自动 push | 只在本地 commit，由用户决定何时 push |

**Commit 格式**:
```
[类型] 简要描述

详细说明这次改动的内容和原因
```

类型: `feat` / `fix` / `refactor` / `exp` / `data` / `eval`

### 2. 科研启动-执行 — 任务执行

触发词: `科研启动-执行`

**核心原则**:
- **从原始需求出发** — 理解用户真正目标，不急于写代码
- **最小完整方案** — 只围绕明确目标，不擅自扩展
- **链路检查** — 输出方案前检查：输入 → 处理流程 → 状态变化 → 输出 → 上下游影响
- 无法验证的部分标注"假设"和"未验证前提"

### 3. 科研启动-检查 — 算法审查

触发词: `科研启动-检查`

**5 个 Agent 分工**:

| Agent | 职责 |
|-------|------|
| Agent 1 | 算法流程审查（逐步追踪输入/输出/逻辑转换） |
| Agent 2 | 边界与异常分析（空值、越界、shape/dtype） |
| Agent 3 | 数值与计算正确性（公式、loss、梯度流） |
| Agent 4 | 性能与效率审查（缓存、并行化、内存） |
| Agent 5 | 对比与一致性检查（代码 vs 论文/伪代码） |

**审查方法**: 逐段阅读 → 逐段思考 → 逐段质疑（理解 → 判断 → 结论）

**输出**: `ISSUES.md`（在项目根目录），逐条展示问题并询问用户分类

## 使用场景

| 场景 | 触发词 |
|------|--------|
| 开始科研编码，需要进度管理和自动 commit | 科研启动 |
| 有明确的代码任务需要执行 | 科研启动-执行 |
| 代码写完了，需要全面审查算法正确性 | 科研启动-检查 |
